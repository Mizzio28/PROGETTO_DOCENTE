Analisi: LAZY vs EAGER

Confronto tra le strategie di fetch JPA/Hibernate usate per caricare Torneo → Squadre → Giocatori, con dati reali misurati.

La differenza

LAZY (usato nel progetto) EAGER
Quando carica i dati collegati Solo quando il codice accede esplicitamente alla collezione Sempre, ad ogni query che carica l'entità
Disattivabile caso per caso? Sì — di default costa 0 query No — costo fisso, incondizionato
Esempio (Squadra.java) fetch = FetchType.LAZY fetch = FetchType.EAGER 
@OneToMany(mappedBy = "squadra", cascade = CascadeType.ALL, fetch = FetchType.LAZY)
private List<Giocatore> giocatori = new ArrayList<>();

Tutte le associazioni del progetto (Torneo, Squadra, Giocatore, Partita, Commento) sono mappate LAZY.

Dati: LAZY vs EAGER

Scenario Servono squadre/giocatori? LAZY (progetto) EAGER (simulato)

Sulla homepage (che non mostra mai squadre/giocatori), EAGER costa 11 query/run contro 1: il costo si paga sempre, anche quando il dato non serve. Per questo il progetto usa LAZY.

Il problema N+1

Quando il codice scorre una collezione lazy e per ogni elemento accede a un'altra collezione lazy (es. per ogni Squadra del torneo, i suoi Giocatori), Hibernate esegue una query aggiuntiva per ogni elemento: non 1 query, ma N+1.

Nel progetto si evita mirando il fetch dove serve, non rendendo tutto EAGER: TorneoRepository.findWithSquadreById usa @EntityGraph(attributePaths = {"squadre"}) per caricare torneo e squadre in un'unica query nella pagina di dettaglio torneo, lasciando i giocatori lazy — quella pagina non li mostra mai.
